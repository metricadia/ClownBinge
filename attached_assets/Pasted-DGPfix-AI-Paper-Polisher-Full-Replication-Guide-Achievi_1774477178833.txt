1. System Overview

The AI detection defeat pipeline works in a detect → target → rewrite → re-detect → loop cycle:

User uploads .docx / .txt
         ↓
  mammoth parses DOCX → structured plain text
         ↓
  parseAcademicSections() splits into major sections
  (References / Bibliography → flagged isProtected = true, NEVER touched)
         ↓
  ZeroGPT scans each section → returns overallScore (0–100%) + flaggedSentences[]
         ↓
  If score > targetScore (default 20%):
    For each flaggedSentence:
      Claude 3.5 Sonnet rewrites it using document-type-specific prompt
    Replace originals in text → re-scan with ZeroGPT
    Repeat up to 10 times or until score ≤ target
         ↓
  Save cleaned content to DB, update aiDetectionScore
         ↓
  User exports APA 7 formatted .docx
Internal codename: MajorBrain (log prefix: [MajorBrain])

Real-world results: Consistently achieves 5–18% AI detection on ZeroGPT after 2–4 cleanup iterations. The system targets 20% but in practice lands at 5–15%.

DOCX → Structured Plain Text Conversion

This is critical. Mammoth's default extractRawText() loses all heading structure. We use convertToHtml() and then apply a custom regex pipeline to preserve section headings as standalone lines:

function mammothHtmlToStructuredText(html: string): string {
  return html
    // Headings → standalone lines with surrounding double newlines
    .replace(/<h[1-3][^>]*>([\s\S]*?)<\/h[1-3]>/gi, '\n\n$1\n\n')
    // Paragraphs → double newline separated
    .replace(/<p[^>]*>([\s\S]*?)<\/p>/gi, '\n\n$1\n\n')
    // List items → newline + bullet
    .replace(/<li[^>]*>([\s\S]*?)<\/li>/gi, '\n• $1')
    // Line breaks
    .replace(/<br\s*\/?>/gi, '\n')
    // Strip inline formatting (keep text content)
    .replace(/<\/?(?:strong|b|em|i|u|s|span|a|sup|sub)[^>]*>/gi, '')
    // Strip remaining HTML tags
    .replace(/<[^>]+>/g, '')
    // Decode HTML entities
    .replace(/&amp;/g, '&')
    .replace(/&lt;/g, '<')
    .replace(/&gt;/g, '>')
    .replace(/&quot;/g, '"')
    .replace(/&#39;/g, "'")
    .replace(/&nbsp;/g, ' ')
    .replace(/&#8220;|&#8221;/g, '"')
    .replace(/&#8216;|&#8217;/g, "'")
    .replace(/&#8212;/g, '—')
    .replace(/&#8211;/g, '–')
    // Collapse 3+ newlines to double newline
    .replace(/\n{3,}/g, '\n\n')
    .trim();
}

Character Limit Handling (CRITICAL)

ZeroGPT has a ~50,000 character API limit. The system automatically chunks large texts:

function splitIntoChunks(text: string, maxChars: number = 45000): string[] {
  if (text.length <= maxChars) return [text];
  const chunks: string[] = [];
  const paragraphs = text.split(/\n\n+/);
  let currentChunk = '';
  for (const para of paragraphs) {
    if (para.length > maxChars) {
      // Oversized single paragraph: split by sentences
      if (currentChunk) { chunks.push(currentChunk.trim()); currentChunk = ''; }
      const sentences = para.match(/[^.!?]+[.!?]+/g) || [para];
      for (const sentence of sentences) {
        if (currentChunk.length + sentence.length > maxChars) {
          if (currentChunk) chunks.push(currentChunk.trim());
          currentChunk = sentence;
        } else {
          currentChunk += sentence;
        }
      }
      continue;
    }
    if (currentChunk.length + para.length + 2 > maxChars) {
      chunks.push(currentChunk.trim());
      currentChunk = para;
    } else {
      currentChunk += (currentChunk ? '\n\n' : '') + para;
    }
  }
  if (currentChunk) chunks.push(currentChunk.trim());
  return chunks;
}
Multi-Chunk Score Combination

When a text is split into multiple chunks, scores are weighted by word count:

const totalWords = chunkResults.reduce((sum, r) => sum + r.wordCount, 0);

HTML Stripping (Before Sending to ZeroGPT)

The stored section content may contain HTML. Strip it before detection:

export function stripHtmlForDetection(html: string): string {
  return html
    .replace(/<\/p>/gi, '</p>\n\n')
    .replace(/<br\s*\/?>/gi, '\n')
    .replace(/<\/h[1-6]>/gi, '</h>\n\n')
    .replace(/<\/div>/gi, '</div>\n\n')
    .replace(/<[^>]*>/g, ' ')
    .replace(/[ \t]+/g, ' ')
    .replace(/\n\n\n+/g, '\n\n')
    .trim();
}

9. The Rewrite Engine — Full Prompt Architecture

File: server/academic-cleanup-service.ts → buildParaphrasePrompt()

Claude API Call Parameters

const response = await anthropic.messages.create({
  model: "claude-sonnet-4-6",   // Claude 3.5 Sonnet — the important choice
  max_tokens: 600,               // Per-sentence rewrite, so 600 is generous
  temperature: 0.85,             // HIGH temperature = more varied/creative = less AI-detectable
  messages: [{ role: "user", content: prompt }],
});
Why temperature 0.85? Lower temperatures make Claude predictable and repetitive — which is exactly what AI detectors look for. High temperature forces creative variation in word choice and sentence structure, which increases perplexity (the statistical measure detectors use).

Why claude-sonnet-4-6 and not Haiku? Haiku tends to produce more formulaic rewrites. Sonnet has the vocabulary range and contextual understanding to make genuinely good sentence-level changes without introducing factual errors.

Sentence Replacement in Text

After rewriting, each flagged sentence is replaced back using regex:

function replaceSentences(plainText: string, replacements: Map<string, string>): string {
  let updated = plainText;
  Array.from(replacements.entries()).forEach(([original, paraphrased]) => {
    // Escape special regex chars in the original sentence
    const escapedOriginal = original.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
    const regex = new RegExp(escapedOriginal, 'g');
    updated = updated.replace(regex, paraphrased);
  });
  return updated;
}
10. The Auto-Cleanup Loop (MajorBrain)

File: server/academic-cleanup-service.ts → autoCleanAcademicSection()

This is the core algorithm. It loops until the AI score falls below the target or it runs out of attempts.

Function Signature

async function autoCleanAcademicSection(
  content: string,          // Section text (may contain HTML)
  targetScore: number = 20, // Stop when score ≤ this (use 15 to be safer)
  maxAttempts: number = 10, // Maximum iteration cap
  docType: DocumentType = "undergrad"
): Promise<CleanupResult>
Full Algorithm

1. Strip HTML → plain text (stripHtmlForDetection)
2. Scan with ZeroGPT → get initialScore
3. If initialScore ≤ targetScore → return immediately (already passing)
4. Loop while score > targetScore AND attemptNumber < maxAttempts:
   a. Scan with ZeroGPT → get flaggedSentences[]
   b. If flaggedSentences is empty → break (nothing more to fix)
   c. For each flaggedSentence:
      - Call Claude Sonnet 3.5 with document-type prompt
      - Store original → rewritten mapping
   d. Apply all replacements to the text
   e. Re-scan with ZeroGPT → get newScore
   f. Record attempt metadata
   g. Update currentScore = newScore
   h. If currentScore ≤ targetScore → break (success)
   i. EARLY EXIT: if last 2 attempts show no score improvement → break (stuck)
5. Return CleanupResult
Early Exit Logic (Important)

if (attempts.length >= 2) {
  const lastTwo = attempts.slice(-2);
  if (lastTwo[1].scoreBeforeParaphrase >= lastTwo[0].scoreBeforeParaphrase) {
    console.log("[MajorBrain] Score not improving. Breaking loop.");
    break;
  }
}
This prevents infinite loops when a section's structure makes further improvement impossible.

Return Type

interface CleanupResult {
  success: boolean;           // true if finalScore ≤ targetScore
  originalScore: number;      // Score before any cleanup
  finalScore: number;         // Score after cleanup
  attempts: CleanupAttempt[]; // Per-attempt metadata
  totalAttempts: number;
  cleanedContent: string;     // The rewritten plain text
  message: string;            // Human-readable summary
}
interface CleanupAttempt {
  attemptNumber: number;
  scoreBeforeParaphrase: number;
  flaggedSentencesCount: number;
  paraphrasedSentencesCount: number;
}
Save Logic (Routes Layer)

Only save cleaned content if it's actually better:

if (newScore < currentScore) {
  await storage.updateAcademicSection(section.id, {
    content: result.cleanedContent,

14. Why It Works — Detection Defeat Theory

Understanding the theory is essential for tuning the system to hit ≤15% reliably.

How AI Detectors Work

Detectors like ZeroGPT, GPTZero, and Originality.ai primarily measure two statistical properties of text:

1. Perplexity — How predictable each word is given the words before it. AI tends to choose the statistically most probable next word. Human writers make unexpected choices.

2. Burstiness — How varied sentence lengths are. AI writes very consistently-lengthed sentences. Humans write short punchy sentences followed by much longer, more complex constructions.

Why Standard AI Rewrites Fail

If you ask an LLM to "make this sound more human," it defaults to:

Adding transition words ("Furthermore", "Moreover") → these are the highest-probability AI tokens
Creating parallel structures → AI signature pattern
Using generic hedging language → AI signature pattern
Producing sentences of consistent length → low burstiness = high AI detection
The MajorBrain Counter-Strategy

Kill Words enforce anti-probability vocabulary — By explicitly banning the tokens LLMs most commonly produce, we force the model to choose lower-probability alternatives. This directly increases perplexity scores.

Document-type voice injection — Rather than generic "sound human," we give the model a specific person to emulate (community practitioner, senior scholar, person at kitchen table). Specific personas produce specific word choices with specific idiosyncrasies — much harder to detect.

Preserve word count — If rewritten sentences are dramatically shorter or longer, ZeroGPT's sentence-level analysis patterns shift. Staying within ±5 words keeps the text coherent and detection-consistent.

No em dashes — AI models have a measurable over-reliance on em dashes as a stylistic choice. Banning them removes a reliable AI signal.

Citation preservation — Citations like (Author, Year) are human reference artifacts. Preserving them exactly maintains a key human-authorship signal.

Concrete over abstract (grant writing) — Detectors are tuned to recognize abstract institutional language. Replacing "institutional systems" with "American schools, healthcare, and public agencies" shifts vocabulary to the concrete, specific language of human practitioners.

Temperature 0.85 — High enough to produce genuinely varied outputs between runs, preventing the model from falling back into predictable output patterns.

To Target ≤15% Specifically

Change the auto-cleanup call to use targetScore: 15:

Or pass it explicitly from the frontend:

fetch(`/api/academic-sections/${id}/auto-cleanup`, {
  method: "POST",
  body: JSON.stringify({ targetScore: 15 })
})
If Results Plateau Above Target

Some sections are structurally difficult to get below 20% in one pass. Strategies:

Run multiple passes — call auto-cleanup again on the same section after it finishes
Increase max attempts — change maxAttempts from 10 to 15 (more API cost)
Lower the detection threshold — the early-exit logic breaks if two consecutive attempts show no improvement; this can be loosened by requiring 3 non-improving attempts
Manual editing — have the user manually edit the hardest sentences, then re-run
16. Known Edge Cases & Guardrails

What the System Will NOT Do

Touch References/Bibliography — isProtected = true sections are completely skipped at every layer. The API routes return immediately. The cleanup function is never called. Citations are sacred.
Process sections under 100 characters — Too short for meaningful detection or rewriting.
Save worse versions — If cleanup doesn't improve the score, the original content stays. Only the scan score is saved.
Known Failure Modes

Highly formulaic sections (e.g., methods tables, numbered lists) — These often plateau at 25–35% because the structure itself is AI-detectable. The early-exit logic prevents wasted API calls. Manual editing is the solution.

Technical PhD content with dense terminology — ZeroGPT sometimes flags technical jargon as AI even when human-written. Lowering the acceptable target to 25–30% for PhD technical sections is pragmatic.

Grant budgets and timelines — Formatted tabular content doesn't parse cleanly into sentences. parseAcademicSections will capture them as content but they won't rewrite well. Consider making these sections user-editable only.

Repeated re-runs on the same section — Each run improves from the last cleaned version. Running auto-cleanup a second time on a section that landed at 18% will often push it to 8–12%.

API Rate Limits

ZeroGPT (RapidAPI): Free tier is limited (check your RapidAPI dashboard). Each auto-cleanup run consumes: 1 initial scan + up to 10 re-scans + 1 per-attempt rescan = potentially 20+ API calls per section. For a 6-section paper, that's up to 120 ZeroGPT calls.

Anthropic (Claude): Each flagged sentence costs one API call. A 10-sentence section with 8 flagged sentences over 3 attempts = 24 Claude calls. Claude's per-call cost at Sonnet pricing is ~$0.003 per call. Budget accordingly.
